# Exercise: What Should You Actually Measure?

**Type:** Group  
**Motivation:** Monitoring is only useful if you measure the right things. This exercise pushes you to think critically about what "healthy" looks like for your specific deployment — not just collect metrics because you can.  
**Exam:** Can be used for exam project

---

## Task

Your group has two Azure VMs:
- **VM 1:** Nginx (reverse proxy / load balancer)
- **VM 2:** Your backend application
- If you have more it is fine, but this task deals with 2

For each layer, identify **at least 2 meaningful metrics** you want to monitor. Then implement at least one of them.

### Step 1 — Brainstorm metrics for each layer

Think about what could go wrong or degrade silently. Consider metrics in these categories:

| Category | Examples to inspire you |
|---|---|
| Availability | Is the service up? Is nginx actually proxying requests? |
| Traffic | How many requests per second? Which routes are hit most? |
| Errors | HTTP 4xx and 5xx rates — are errors spiking? |
| Latency | How long do requests take? Is one endpoint slow? |
| Resources | CPU, memory, disk on each VM — are you running out? |
| Business | How many recipes created today? Login attempts? Search queries? |

Write down your chosen metrics as a group (a short list in your `README.md` or a `monitoring/metrics.md` file is fine).

### Step 2 — Implement one metric 

Look at your current Prometheus/Grafana setup. Pick one metric from your brainstorm and add it.

Good candidates if you need inspiration:
- `nginx_http_requests_total` via [nginx-prometheus-exporter](https://github.com/nginxinc/nginx-prometheus-exporter)
- A custom application counter for a domain-specific event (e.g. `recipes_created_total`)
- Node exporter's `node_filesystem_avail_bytes` to track disk space on the backend VM

### Step 3 — Add an alert threshold (optional but recommended)

In Grafana, set a threshold or alert on the metric you added. Ask yourself: at what value should your team be notified?

---

## Deliverable

- A short written list of the metrics your group decided to track and **why** each one matters for your specific application
- At least one new metric visible in your Grafana dashboard
- The Grafana dashboard JSON exported and committed to your repo

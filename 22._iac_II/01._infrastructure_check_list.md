# Infrastructure Checklist

Use this checklist to verify that your group's IaC setup is complete and working. Every group member should be able to run through the full cycle independently — not just one person.

---

## Prerequisites

- [ ] Azure CLI (`az`) is installed on your machine
- [ ] You are logged in to Azure (`az login`)
- [ ] You have a personal SSH key pair generated
- [ ] Your public SSH key path is known and set correctly in the script
- [ ] You have access to your group's GitHub repository
- [ ] You have a `CR_PAT` (GitHub Personal Access Token with `read:packages` scope) ready

---

## Repository Setup

- [ ] Your group has an `infrastructure/` folder at the root of the project
- [ ] The infrastructure scripts are committed and pushed to the master branch
- [ ] The repository has a `infrastructure/README.md` (or a section in the main README) explaining:
  - What prerequisites are needed
  - How to run the setup script
  - How to run the teardown script

---

## Part 1 — Single VM

- [ ] `infrastructure/create_vm.sh` exists and is executable
- [ ] The script creates a resource group on Azure
- [ ] The script creates a single VM (e.g. `Standard_B1s`)
- [ ] The necessary ports are opened (80, 443, and your app port)
- [ ] The script installs dependencies on the VM (Docker, Docker Compose, or your stack equivalent)
- [ ] The script sets up GitHub Actions secrets (e.g. VM IP, SSH key)
- [ ] After running the script, the GitHub Actions workflow runs and deploys the application
- [ ] The application is accessible in the browser via the VM's public IP

---

## Part 2 — Two-VM Setup

- [ ] `infrastructure/create_two_vms.sh` exists and is executable
- [ ] The script provisions two VMs in the same resource group and virtual network:
  - [ ] `nginx-vm` — public-facing, handles traffic on port 80/443
  - [ ] `backend-vm` — internal only, not directly reachable from the internet
- [ ] The Nginx VM is configured to proxy requests to the backend VM's **private IP**
- [ ] The backend VM's network security group (NSG) only allows traffic from the nginx VM on the app port
- [ ] After running the script, the GitHub Actions workflow deploys the application
- [ ] The application is accessible via the nginx VM's public IP
- [ ] The backend VM is **not** directly reachable from the browser

---

## Teardown

- [ ] `infrastructure/azure-teardown.sh` (or equivalent) exists and is executable
- [ ] Running the teardown script removes **all** resources from Azure (resource group deleted)
- [ ] After teardown, no resource groups remain in the Azure portal for this project

---

## Full Cycle — Everyone in the Group

This is the most important part. Each group member must be able to do this independently:

- [ ] **Member 1**: Run setup → confirm deployment works → run teardown
- [ ] **Member 2**: Run setup → confirm deployment works → run teardown
- [ ] **Member 3**: Run setup → confirm deployment works → run teardown
- [ ] *(repeat for all members)*

No changes to the scripts should be needed between runs. If someone has to modify the script to make it work on their machine, fix it so it works for everyone.

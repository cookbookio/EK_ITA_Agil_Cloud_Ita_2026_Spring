## Part 1 — Single VM setup script

Create an `infrastructure/` folder at the root of your project and add a shell script that provisions a **single Azure VM** running your entire application.

The script should at minimum:

- Create a resource group
- Create a VM (use a sensible size, e.g. `Standard_B1s`)
- Open the necessary ports (e.g. 80, 443, and your app port)
- SSH into the VM and install dependencies (Docker, Docker Compose, or whatever your stack needs)
- Github secrets setup or update

```
infrastructure/
└── create_vm.sh
```

Think about what should be **parameterized** (resource group name, VM name, location, your public SSH key path) versus hardcoded.

> **DevOps angle**: This script should be idempotent where possible. What happens if someone runs it twice? Document any assumptions or prerequisites at the top of the script.

---

## Part 2 — Two-VM setup: separate Nginx and backend

Extend your infrastructure to a two-VM setup. This is a new feature branch in your project. 

```
infrastructure/
├── create_vm.sh          # from Part 1 (single VM)
└── create_two_vms.sh     # new script for the split setup
```

The two VMs should be:

| VM | Role |
|----|------|
| `nginx-vm` | Nginx reverse proxy — public-facing, handles incoming traffic on port 80/443 |
| `backend-vm` | Your application backend — only reachable from the nginx VM, not directly from the internet |

Your script should:

1. Provision both VMs in the same resource group and virtual network so they can communicate privately
2. Configure the **nginx VM** with an Nginx config that proxies requests to the **backend VM's private IP**
3. Configure the **backend VM** to run your application (without exposing it publicly)
4. Lock down the backend VM's network security group so only the nginx VM can reach the app port

> **DevOps angle**: This split mirrors a production-grade setup. Document *why* you made the networking decisions you did — this will strengthen your exam report.

---

## Hints

- Use `az vm create` with `--public-ip-address ""` on the backend VM to avoid assigning a public IP
- Use `az network nsg rule create` to restrict inbound traffic on the backend VM
- `az vm show -d` can retrieve the private IP of a VM once it is created — useful for templating your Nginx config
- Use `ssh -o StrictHostKeyChecking=no` and `scp` or `cloud-init` to push config files to the VMs from your script
